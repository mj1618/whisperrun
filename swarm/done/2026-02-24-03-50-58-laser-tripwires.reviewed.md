# Task: Laser Tripwires

## Overview

The game currently has two threat types: **guards** (mobile, vision-cone-based) and **security cameras** (fixed position, sweeping vision cone, alert nearest guard). Both are "active" threats that watch for the Runner. Laser tripwires add a **passive environmental threat** — beams that the Runner must time and dodge, creating classic heist tension moments.

Lasers are the most iconic heist obstacle. They create the "step carefully through the laser grid" moments that both players will recognize instantly. For the Whisper, lasers are fully visible on the blueprint, creating clear communication moments: "Wait... wait... NOW, go!" For the Runner, lasers are visible within fog-of-war range and pulse on/off in patterns, requiring timing and patience.

This feature scales naturally with difficulty: Casual gets 0-1 lasers, Standard gets 1-2, Hard gets 2-3. Lasers also combine with existing mechanics — a guard alert near a laser corridor creates panic as the Runner has to dodge both.

### 1. Laser Tripwire Game Logic (`/src/game/lasers.ts` — CREATE)

Create a new module defining laser tripwire types and logic:

```typescript
export interface LaserTripwire {
  id: string;
  /** Emitter position (tile coordinates — the wall tile the laser originates from) */
  x1: number;
  y1: number;
  /** Receiver position (tile coordinates — the wall tile the laser ends at) */
  x2: number;
  y2: number;
  /** Whether the laser is currently active (beam is on) */
  active: boolean;
  /** Pattern timing: beam is on for `onDurationMs`, off for `offDurationMs` */
  onDurationMs: number;
  offDurationMs: number;
  /** Offset into the cycle at game start (so not all lasers pulse in sync) */
  phaseOffsetMs: number;
}

/**
 * Determine if a laser is currently active based on elapsed time.
 * Lasers cycle: ON for onDurationMs, then OFF for offDurationMs, repeat.
 */
export function isLaserActive(laser: LaserTripwire, elapsedMs: number): boolean {
  const cycleLength = laser.onDurationMs + laser.offDurationMs;
  const adjustedTime = (elapsedMs + laser.phaseOffsetMs) % cycleLength;
  return adjustedTime < laser.onDurationMs;
}

/**
 * Check if the Runner at (rx, ry) is intersecting an active laser beam.
 * The beam is a line between the two wall tiles. The Runner has a small radius (~0.3 tiles).
 * Use point-to-line-segment distance check.
 */
export function isRunnerInLaser(
  laser: LaserTripwire,
  runnerX: number,
  runnerY: number,
  runnerRadius: number = 0.3
): boolean {
  // The beam goes between the inner edges of the two wall tiles
  // (so from just inside x1/y1 wall to just inside x2/y2 wall)
  const bx1 = laser.x1 < laser.x2 ? laser.x1 + 0.9 : laser.x1 + 0.1;
  const by1 = laser.y1 < laser.y2 ? laser.y1 + 0.9 : laser.y1 + 0.1;
  const bx2 = laser.x2 < laser.x1 ? laser.x2 + 0.9 : laser.x2 + 0.1;
  const by2 = laser.y2 < laser.y1 ? laser.y2 + 0.9 : laser.y2 + 0.1;

  // Point-to-segment distance
  const dist = pointToSegmentDist(runnerX, runnerY, bx1, by1, bx2, by2);
  return dist < runnerRadius;
}

function pointToSegmentDist(
  px: number, py: number,
  ax: number, ay: number,
  bx: number, by: number
): number {
  const dx = bx - ax;
  const dy = by - ay;
  const lenSq = dx * dx + dy * dy;
  if (lenSq === 0) return Math.sqrt((px - ax) ** 2 + (py - ay) ** 2);
  let t = ((px - ax) * dx + (py - ay) * dy) / lenSq;
  t = Math.max(0, Math.min(1, t));
  const projX = ax + t * dx;
  const projY = ay + t * dy;
  return Math.sqrt((px - projX) ** 2 + (py - projY) ** 2);
}
```

**Design notes:**
- Lasers are **axis-aligned only** (horizontal or vertical). They span between two wall tiles across a corridor or doorway. This keeps placement and rendering simple.
- Lasers are **on/off cycling**, not always-on. Typical timing: 3s on, 2s off (standard), 4s on / 1.5s off (hard). This gives the Runner windows to pass through.
- Phase offsets stagger the cycles so adjacent lasers don't sync up. The map generator assigns random offsets.
- **Crouching does NOT help** with lasers — they're physical beams, not vision-based. The Runner must wait for the off-cycle to pass.
- **Hiding DOES protect** — if the Runner is in a hide spot, lasers can't detect them (they're inside furniture).

### 6. Laser Detection in GameCanvas (`/src/components/GameCanvas.tsx` — MODIFY)

Add laser-trip detection in the game loop, similar to camera detection. On each frame during the heist phase:

1. Compute elapsed time since heist start.
2. For each laser, check if it's currently active via `isLaserActive()`.
3. If active, check if the Runner intersects the beam via `isRunnerInLaser()`.
4. If Runner is hiding, skip the check (hiding protects from lasers).
5. If the Runner trips a laser, **alert the nearest guard** (same behavior as cameras — `camera_spotted` event equivalent, use a new `"laser_tripped"` event type).

```typescript
import { isLaserActive, isRunnerInLaser } from "@/game/lasers";

// In the game loop, after camera detection and before guard tick:
const elapsedMs = now - (gameState.heistStartTime ?? gameState.startTime);
const laserTrippedGuardId: string | null = null;

for (const laser of gameState.lasers ?? []) {
  if (!isLaserActive(laser, elapsedMs)) continue;
  if (gameState.runner.hiding) continue;
  if (isRunnerInLaser(laser, gameState.runner.x, gameState.runner.y)) {
    // Trip! Alert nearest idle guard
    // Use the same nearest-guard logic as camera detection
    // Record event
    eventRecorder.record("laser_tripped", {
      x: gameState.runner.x,
      y: gameState.runner.y,
    });
    // Play alarm sound
    audioEngine?.playLaserAlarm();
    break; // Only trip one laser per frame
  }
}
```

The alert-nearest-guard logic should reuse the same pattern as camera detection (find nearest guard in patrol/returning state, set to suspicious with last-known at laser position). Look at the existing camera detection code in GameCanvas and mirror that pattern.

**Add a cooldown** (4 seconds, matching noise detection cooldown) so the Runner can't trigger the same laser repeatedly in quick succession. Use a module-level `lastLaserTripTime` variable.

### 8. Laser Rendering — Runner View (`/src/game/runner-view.ts` or `/src/engine/renderer.ts` — MODIFY)

Render lasers within the Runner's fog-of-war. Lasers should be visible when within the Runner's vision radius (they emit light, so they're visible even in fog).

**Visual design:**
- **Active laser beam:** A thin red line (2-3px) with a glow effect (wider, translucent red) between the two wall-mounted emitters. Animate a subtle pulse (opacity oscillation between 0.7 and 1.0) to make it feel alive.
- **Inactive laser (off-cycle):** Show the two emitter boxes on the walls (small dark squares with a dim red dot) but no beam. This helps the Runner see where lasers are even when they're off.
- **Emitter boxes:** Small 6x6px squares on the wall tiles at each end, with a bright red dot (3px) when active, dim red when inactive.

Rendering approach — add a `drawLasers` function to the renderer or to runner-view:

```typescript
function drawLasers(
  ctx: CanvasRenderingContext2D,
  lasers: LaserTripwire[],
  elapsedMs: number,
  camera: Camera,
  tileSize: number,
  visibilityMask?: boolean[][] // fog of war
): void {
  for (const laser of lasers) {
    const active = isLaserActive(laser, elapsedMs);

    // Convert tile coords to screen coords
    const sx1 = (laser.x1 + 0.5) * tileSize - camera.x;
    const sy1 = (laser.y1 + 0.5) * tileSize - camera.y;
    const sx2 = (laser.x2 + 0.5) * tileSize - camera.x;
    const sy2 = (laser.y2 + 0.5) * tileSize - camera.y;

    // Check if either end is visible in fog of war
    // (Lasers glow, so they're visible if either endpoint is in range)
    // ... fog check ...

    if (active) {
      // Glow layer (wider, translucent)
      ctx.strokeStyle = "rgba(255, 40, 40, 0.3)";
      ctx.lineWidth = 8;
      ctx.beginPath();
      ctx.moveTo(sx1, sy1);
      ctx.lineTo(sx2, sy2);
      ctx.stroke();

      // Core beam (thin, bright)
      const pulse = 0.7 + 0.3 * Math.sin(elapsedMs * 0.005);
      ctx.strokeStyle = `rgba(255, 0, 0, ${pulse})`;
      ctx.lineWidth = 2;
      ctx.beginPath();
      ctx.moveTo(sx1, sy1);
      ctx.lineTo(sx2, sy2);
      ctx.stroke();
    }

    // Emitter boxes on both ends
    const dotColor = active ? "#FF0000" : "#660000";
    const boxSize = 6;
    for (const [sx, sy] of [[sx1, sy1], [sx2, sy2]]) {
      ctx.fillStyle = "#333";
      ctx.fillRect(sx - boxSize / 2, sy - boxSize / 2, boxSize, boxSize);
      ctx.fillStyle = dotColor;
      ctx.beginPath();
      ctx.arc(sx, sy, 2, 0, Math.PI * 2);
      ctx.fill();
    }
  }
}
```

```typescript
playLaserAlarm(): void {
  if (this.muted || !this.ctx) return;
  const now = this.ctx.currentTime;

  // Sharp zap: short noise burst
  const noise = this.ctx.createOscillator();
  noise.type = "sawtooth";
  noise.frequency.setValueAtTime(800, now);
  noise.frequency.exponentialRampToValueAtTime(200, now + 0.15);

  const gain = this.ctx.createGain();
  gain.gain.setValueAtTime(0.3, now);
  gain.gain.exponentialRampToValueAtTime(0.01, now + 0.2);

  noise.connect(gain);
  gain.connect(this.masterGain ?? this.ctx.destination);
  noise.start(now);
  noise.stop(now + 0.2);

  // Follow-up beep
  const beep = this.ctx.createOscillator();
  beep.type = "sine";
  beep.frequency.setValueAtTime(1200, now + 0.1);

  const beepGain = this.ctx.createGain();
  beepGain.gain.setValueAtTime(0.2, now + 0.1);
  beepGain.gain.exponentialRampToValueAtTime(0.01, now + 0.3);

  beep.connect(beepGain);
  beepGain.connect(this.masterGain ?? this.ctx.destination);
  beep.start(now + 0.1);
  beep.stop(now + 0.3);
}
```

## Key Design Decisions (Already Made)

1. **Axis-aligned beams only** — Lasers span horizontally or vertically between wall tiles. No diagonal beams. This keeps collision detection simple (point-to-line-segment on axis-aligned lines) and placement intuitive.

2. **On/off cycling, not always-on** — Permanent lasers would be hard walls. Cycling lasers create timing puzzles that require Runner patience and Whisper communication. The off-window is generous enough (1.5-2.5s) to pass through without frame-perfect timing.

3. **Alert nearest guard (same as cameras)** — Tripping a laser doesn't instantly lose the game. It alerts the nearest idle guard, who investigates. This matches the "polite consequences" tone — the Runner has a chance to hide before the guard arrives.

4. **Client-side active/inactive computation** — The laser timing is deterministic from the timing parameters and elapsed time. No need to sync active state through Convex — both clients compute it identically from the same heist start time.

5. **No lasers on Casual difficulty** — Keep Casual approachable. Lasers are a "step up" mechanic for Standard and Hard, giving experienced players more challenge without overwhelming newcomers.

6. **Hiding protects from lasers** — If the Runner is in a hide spot, they're inside furniture and the beam can't reach them. This is consistent with how hiding protects from guard vision and cameras.

7. **4-second trip cooldown** — Prevents spam-alerting guards by standing in a laser beam. One trip, one alert, then a cooldown before it can trigger again.

8. **Corridor/doorway placement** — Lasers are most interesting in chokepoints where the Runner *must* pass through them (can't just walk around). The map generator places them across 1-2 tile wide passages.

9. **Visible emitter boxes even when beam is off** — The Runner can see the wall-mounted emitters within fog-of-war range, giving a "heads up" that a laser is here even during the off-cycle. This supports tactical play over surprise.

10. **Phase offsets for multiple lasers** — When a map has 2-3 lasers, they don't all pulse in sync. Random phase offsets create varied timing patterns that require the Runner to observe each laser independently.
